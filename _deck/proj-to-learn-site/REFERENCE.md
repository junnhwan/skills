# REFERENCE — proj-to-learn-site

Detailed recipe. Read after SKILL.md. All paths relative to the copied site (`<proj>/docs/site`).

## 0. Get the reference app

If `learn-claude-code/web` is not already on disk:
```bash
git clone https://github.com/shareAI-lab/learn-claude-code
# the app is at learn-claude-code/web
```

## 1. Scaffold (mechanical)

```bash
bash <skill>/scripts/bootstrap-site.sh <learn-claude-code>/web <proj>/docs/site
cd <proj>/docs/site && npx next dev -p 3000
```
The script copies the app, **neuters `predev`/`prebuild`** in package.json (they call `extract-content.ts` which scans `sNN_*/code.py` and would **overwrite your hand-authored data on every `npm run dev`/`build`**), and runs `npm install`.

> Why hand-author instead of using their extract? `extract-content.ts` is **Python-specific** (parses `code.py` for classes/funcs/tools). For non-Python projects it produces nothing useful. Hand-authoring `generated/*.json` + `scenarios/*.json` is language-agnostic and lets you curate the best excerpts.

## 2. Data schemas (from `src/types/agent-data.ts`)

**AgentVersion** (one per chapter, in `generated/versions.json` → `versions[]`):
```ts
{ id, filename, title, subtitle, loc:number, tools:string[], newTools:string[],
  coreAddition, keyInsight, layer:"tools"|"planning"|"memory"|"concurrency"|"collaboration",
  classes:{name,startLine,endLine}[], functions:{name,signature,startLine}[],
  source:string, images:[] }
```
- `source` = a **real, curated code excerpt** shown in the 源码 tab (escape `"`→`\"`, newline→`\n` in JSON; backticks are fine). Keep it focused (~20-40 lines), quote-light.
- `classes` ≈ the project's key types/structs/interfaces; `functions` ≈ key funcs/methods.

**VersionDiff** (`generated/versions.json` → `diffs[]`): `{from,to,newClasses[],newFunctions[],newTools[],locDelta}`.

**DocContent** (`generated/docs.json[]`): `{version, locale:"en"|"zh"|"ja", title, content}` — `content` is markdown (rendered by remark/rehype with GFM + highlight.js).

**Scenario** (`scenarios/<id>.json`): `{version,title,description,steps:SimStep[]}`.
**SimStep**: `{type:"user_message"|"assistant_text"|"tool_call"|"tool_result"|"system_event", content, annotation, toolName?, toolInput?}`.

## 4. Visualization — use MechanismFlow, do NOT reuse Claude-Code viz shells

> ⚠ **Critical (was a silent defect).** The reference app ships 20 viz components (`s01-agent-loop.tsx` … `s20-comprehensive.tsx`) whose **content is hard-coded to Claude Code's mechanisms**. Reusing them by remapping `index.tsx` (the old § 4 advice) only borrows the *animation shell* — the nodes/text/steps still describe Claude Code, NOT the user's project. The viz is the site's headline feature; content mismatch = headline broken. **Do not reuse them as-is.**

### 4.1 ✅ Correct: author custom viz via `MechanismFlow` (data-driven template)

The app already ships a **data-driven template** at `src/components/visualizations/shared/mechanism-flow.tsx` — `MechanismFlow`. It renders an animated, step-through node/edge graph where **you supply the data**, so the graph matches the project's actual mechanism. This is the supported way to make every chapter's viz correct.

**Authoring one custom viz = one small data file.** Create `src/components/visualizations/<proj>-<mechanism>-flow.tsx`:

```tsx
"use client";
import { MechanismFlow } from "./shared/mechanism-flow";

export default function MyMechanismFlow({ title }: { title?: string }) {
  return (
    <MechanismFlow
      title={title}
      fallbackTitle="<一句话机制名>"
      viewBox="0 0 760 400"
      nodes={[
        { id: "start", label: "入口", x: 80, y: 60, kind: "start", appearsAt: 0 },
        { id: "decide", label: "风险判断", x: 280, y: 160, kind: "decision", appearsAt: 1 },
        { id: "store", label: "落盘", x: 480, y: 260, kind: "store", appearsAt: 2 },
        { id: "end", label: "回填", x: 680, y: 360, kind: "end", appearsAt: 3 },
      ]}
      edges={[
        { id: "e1", from: "start", to: "decide", label: "提交", appearsAt: 1 },
        { id: "e2", from: "decide", to: "store", label: "通过", appearsAt: 2 },
        { id: "e3", from: "store", to: "end", label: "完成", appearsAt: 3 },
      ]}
      steps={[
        { title: "步骤1", description: "...(引用 file:line)", focus: ["start","decide","e1"] },
        { title: "步骤2", description: "...", focus: ["decide","store","e2"] },
        { title: "步骤3", description: "...", focus: ["store","end","e3"] },
      ]}
      footer={["key=val", "不变量=..."]}
    />
  );
}
```

Then in `index.tsx`:
```tsx
s05: lazy(() => import("./bond-subagent-flow")),  // your custom data-driven viz
```

**Data schema** (see `shared/mechanism-flow.tsx` for the source of truth):
- **`nodes`**: `{id, label, x, y, kind?("start"|"process"|"decision"|"store"|"external"|"end"), appearsAt?(step idx when it fades in)}`
- **`edges`**: `{id, from, to, label?, appearsAt?}`
- **`steps`**: `{title, description, focus?(ids to highlight at this step)}`
- **`viewBox`**: SVG canvas; widen height (`0 0 760 440`) for tall graphs. `appearsAt` controls progressive reveal; `focus` drives the per-step highlight.
- `kind` colors: start=blue / process=green / decision=amber / store=purple / external=red dashed / end=slate.

**Coordinate layout tips** (the graph is pure SVG, absolute `x,y`):

**节点尺寸（从 mechanism-flow.tsx 源码）：**
- 矩形节点（process/decision/store）：`width=118px, height=42px`
- 菱形节点（decision）：`size=54×54px`
- label 文字：`fontSize=10px`（矩形）、`9px`（菱形），`fontFamily=monospace`

**安全间距规则（避免重叠/箭头穿刺）：**
- 垂直最小间距：**140px**（节点中心到中心）— 小于这个值文字会重叠
- 水平最小间距：**200px**（分支节点之间）
- 画布留白：上下左右各留 **60-80px** 边距

**Label 文字限制（避免溢出框）：**
- 单行 label：**≤10 个中文字**（或 ≤12 个英文字母）
- 换行（`\n`）：**最多 2 行**，超过 2 行会溢出节点框
- 推荐：简洁动词短语（如 `"LLM Stream"` 而非 `"LLM\nStream\n调用"`）

**appearsAt 可见性问题：**
- 设为 `appearsAt: 2` 或更大 → 前面步骤节点透明度只有 16%（几乎不可见）
- **修复**：重要节点设为 `appearsAt: 0` 或 `1`，让它们始终可见或在第 1 步就出现
- focus 数组：必须包含该步骤要高亮的所有相关节点（包括端点），否则连接的边也不会高亮

**Canvas ~720×400 fits ~6-8 nodes. For more, widen `viewBox` (e.g. `0 0 1120 520` for 9+ nodes) and spread nodes.
- Vertical pipelines: stack `y` at 80/220/360/480 (140px spacing). Branches: same `y`, spread `x` by 200px.
- `MechanismFlow` auto-draws straight/curved edges + arrowheads; you only place node centers.

**Layout recipes** (pick by mechanism shape, reuse the coordinate pattern):
- **Linear pipeline** (request → middleware → handler → store → response): one column, `y` = 60/160/260/360. Good for CLI tools, request paths.
- **Branch + merge** (input → decision → 2-3 parallel paths → merge): main spine at `x=120`; branches fan out at `x=470`/`660`, reconverge at `x=470`. Good for safety policy, dispatch.
- **Layered stack** (L0 → L1 → L2 → L3 → L4 pipeline): vertical column with side stores (e.g. spill store beside L2). Good for context governance, multi-stage processing.
- **DAG / fan-out-fan-in** (plan → parallel nodes → synthesize): left→right flow; parallel nodes same `x` different `y`; one isolation store (e.g. worktree) feeding parallel nodes. Good for orchestration, map-reduce.
- **Lifecycle ring** (4 hooks around a loop): loop center, 4 hook nodes at N/E/S/W. Good for hook systems, event lifecycles.

These are the 5 shapes that cover ~90% of mechanisms. Adapt `viewBox` height to node count; keep ≥80px between nodes to avoid overlap.

### 4.2 When is reuse acceptable?

Reusing a Claude-Code viz shell (`import("./s04-subagent")`) is **only** acceptable when:
1. The project's mechanism genuinely *is* the same shape as that Claude-Code component (rare — usually it isn't), AND
2. You accept the nodes/labels will reference Claude-Code concepts, not the user's project.

For a portfolio/interview-prep site this is almost never acceptable — the viz must depict the user's actual mechanism. **Default to § 4.1.** Treat the 20 shipped components as *fallbacks you can copy and rewrite*, not as content to reuse.

### 4.3 Other custom viz types (advanced, beyond MechanismFlow)

If a mechanism needs something `MechanismFlow` can't express (timelines, FSMs, layered arch), build a new component in `src/components/visualizations/` reusing the shared hooks (`useSteppedVisualization`, `useDarkMode`/`useSvgPalette`) and `shared/step-controls.tsx`. Copy an existing complex component (e.g. `s06-context-compact.tsx` for a layered pipeline) as a skeleton, gut its hard-coded Claude-Code data, and inject the project's data. Keep framer-motion + Tailwind v4 (already in deps).

## 8. 常见可视化布局问题排查

**症状：文字超出节点框**
- **原因**：label 包含超过 10 个中文字，或使用 3 行换行（`\n\n`）
- **修复**：精简 label（如 `"LLM Stream"` 而非 `"LLM\nStream\n调用"`），或拆分成多个节点

**症状：节点几乎不可见（幽灵节点）**
- **原因**：`appearsAt: 2` 或更大，导致前面步骤透明度只有 16%
- **修复**：重要节点设为 `appearsAt: 0` 或 `1`，让它们始终可见

**症状：箭头混乱/穿刺节点框**
- **原因 1**：节点间距过小（<140px 垂直，<200px 水平），`edgePath` 算法无法找到干净路径
- **修复**：增大间距，或调整 `viewBox` 扩大画布
- **原因 2**：focus 数组不包含边的端点，导致边不高亮、路径计算错误
- **修复**：在 `steps[].focus` 中同时包含起点、终点、边的 id

**症状：节点重叠/拥挤**
- **原因**：垂直间距 < 140px（节点高度 42px + 安全距离 ~100px）
- **修复**：重新布局，`y` 坐标按 `80, 220, 360, 480` 间隔（140px）

**症状：整个图表被截断**
- **原因**：节点坐标超出 `viewBox` 范围（如节点 `x=700` 但 `viewBox="0 0 760 400"`）
- **修复**：扩大 `viewBox`（如 `"0 0 1120 520"` 容纳 9+ 节点）

**快速验证 checklist：**
1. 所有 label 是否 ≤2 行且每行 ≤10 中文字？
2. 关键节点是否 `appearsAt ≤ 1`？
3. 节点间距是否 ≥140px（垂直）、≥200px（水平）？
4. `viewBox` 是否留有上下左右各 60-80px 边距？
5. 每个 step 的 focus 数组是否包含该步骤的所有相关节点+边？

## 9. Known residuals (must-fix vs optional)

**⚠️ Must fix — simulator hardcodes all chapter imports.** `src/components/simulator/agent-loop-simulator.tsx` has a `scenarioModules` map with **static `import()` lines for s01..s20**. If your project has fewer chapters (e.g. s01..s08), every import past your last chapter is a **Module not found → 500 error on every page**. **Trim the map to exactly your `VERSION_ORDER` chapters.** (Discovered on BondCode: 4→8 chapter expansion exposed 12 dangling imports.)

Corollary: if you delete unused `scenarios/sNN.json` files (cleanup), you **must** also trim `scenarioModules` to match — otherwise dev/build breaks. The two are coupled.

- The "深入探索" (deep-dive) tab pulls from `execution-flows.ts` + arch components that still reference the original 20-chapter Claude-Code data. Either customize those or hide the tab if the user only needs learn/simulate/code.
- Extra i18n session/viz keys for unused chapters are harmless (not in VERSION_ORDER) — leave or prune.

## 9. Deploy to Cloudflare Pages (static hosting)

The copied app **already has static export configured** — `next.config.ts` ships with `output: "export"`, `images: { unoptimized: true }`, `trailingSlash: true`, and there is **no middleware** and **no `next/image`**. So `next build` produces a fully static `out/` (verified: ~6MB, ~40 prerendered pages, JS/CSS/assets under `_next/static/`). CF Pages serves it directly — no Node runtime needed.

### Method A — wrangler CLI (scriptable, one command)
One-time on the host: `npm i -g wrangler && wrangler login`. Then:
```bash
bash <skill>/scripts/deploy-pages.sh <proj>/docs/site <project-name>
# script runs `npm run build` then `wrangler pages deploy out --project-name <project-name>`
```

### Method B — CF Pages dashboard (git-connect, no CLI)
Connect the repo in the Cloudflare dashboard → Pages → framework preset **"Next.js (Static HTML Export)"** →
- Build command: `npm run build`
- Build output directory: `out`
Push to deploy.

### Hosting many projects under one roof
| Pattern | How | When |
|---|---|---|
| **Subdomain per project** (recommended) | Each project's `docs/site` → its own CF Pages project → `<project>.learn.your.domain`. Bind wildcard `*.learn` in DNS. | Simplest; each project independent, easy to share/link. |
| **One domain + basePath** | One Pages project; each project builds with `basePath: '/<project>'` (set in next.config) and you merge `out/` trees under paths. | Want a single domain. |
| **Unified multi-course app** | Refactor skill to emit a "course module" into one Next app with a top-bar course switcher. | Best UX, most setup. |

For a resume/portfolio use case, **subdomain-per-project** is the sweet spot: free-tier CF Pages hosts dozens, each project deploys independently, and you get clean per-project links.

### Gotchas
- The root `/` redirects to `/en/` via `redirect()` in `src/app/page.tsx` — fine for static (prerenders a redirect). Change to `/zh/` if you want Chinese as the entry locale.
- `_next/static/*` is hashed & immutable; CF caches it aggressively by default — good.
- 404: CF Pages needs the SPA/404 fallback off (it's a static multi-page site; each route has its own `index.html`, so default behavior is correct). The `out/404.html` handles unknown paths.
