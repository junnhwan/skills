# 五层递进结构详解

本文档详细说明每个技术亮点章节的五层递进结构模板。

---

## 第一层：背景痛点

### 目的
让面试官理解你为什么要引入这个技术，证明这不是"为了用而用"。

### 模板
```
### 背景痛点

在系统初期 / 原方案中，[描述原来的做法]。这导致了以下问题：

1. **[痛点1]**：[具体问题] + [量化影响]
2. **[痛点2]**：[具体问题] + [量化影响]

其中最核心的问题是 [突出一个最严重的痛点]，这促使我引入 [技术方案]。
```

### 要点
- 必须有量化数据（"接口响应超过60秒"、"QPS仅263"）
- 原方案必须是合理的初始方案（比如同步处理、直接查库），然后说明为什么不够用
- 痛点要从用户体验、系统稳定性、资源成本多个角度描述

## 第二层：技术选型对比

### 目的
展示技术判断力，证明选型是基于业务场景的分析而非跟风。

### 模板
```
### 技术选型

我对比了以下方案：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 方案A | ... | ... | ... |
| 方案B | ... | ... | ... |
| 最终选择 | ... | ... | ... |

最终选择 [方案X]，核心原因是 [基于业务特点的分析]。
此外还有务实的考量：[技术栈同源 / 已有基础设施复用 / 社区支持]。
```

### 埋钩子技巧
在选型描述中故意留一个可以深挖的点。原则：**只埋自己能接住的钩子**。

| 钩子类型 | 话术示例 | 引导追问方向 |
|----------|---------|------------|
| 反常选型 | "这里我有个看似诡异的选型..." | "为什么这么选？" |
| 故意不展开 | "底层用到了XX机制"（不说细节） | "XX机制是什么？" |
| 主动暴露局限 | "不过这个方案在XX场景下不适用" | "那你会怎么改？" |
| 抬高备选方案 | "方案B在XX方面其实更好" | "那你为什么不选B？" |

## 第三层：具体实现流程

### 目的
证明你真正理解代码的实现细节，不是纸上谈兵。

### 模板
```
### 实现流程

整体流程分为 X 个阶段：

**阶段一：[名称]**
[具体步骤。用文字描述数据如何流转，经过哪些组件]

**阶段二：[名称]**
[具体步骤。包含关键的判断分支]

**数据结构设计：**
| 组件 | Key/结构 | 说明 |
|------|---------|------|
| Redis | `prefix:{identifier}` | ... |
| MySQL | 表名，索引名 | ... |
| MQ | Exchange / Queue / 消息格式 | ... |

**关键设计决策：**
- 为什么用 X 数据结构而不是 Y？
- 为什么在 X 阶段做而不是 Y 阶段？
- 这种选择在什么场景下会变成劣势？
```

### 要点
- **从代码中提取真实细节**，不要凭空编造数据结构
- 每个数据结构的设计都要有理由
- 用伪代码或文字描述关键逻辑，不要贴大段源码

## 第四层：面试 Q&A 模拟

### 目的
预设面试官的所有可能追问，提前准备好完整话术。

### 格式
```
#### Q: [面试官的问题]

**面试官心理：** [具体的、一句话说明面试官真正想考什么]

**回答：**
[第一人称话术，模拟面试现场。像在跟面试官对话，不是在写技术文档]
```

### 面试官心理的写法规范

| 写法 | 示例 | 评价 |
|------|------|------|
| ❌ 太泛 | "考察系统设计思维" | 没有指导意义 |
| ✅ 具体 | "异步任务最怕'死无对证'，后台报错了前台还在傻等。这里考你的异常处理意识" | 有指导意义 |
| ✅ 具体 | "你说你用 Redis，面试官想知道你是不是为了用而用。这里考的是选型判断力" | 有指导意义 |
| ✅ 具体 | "这题本质是在问：单机方案和分布式方案的边界在哪里" | 有指导意义 |

### 深度校准对应表

根据阶段二确认的深度等级，调整 Q&A 的覆盖范围：

| 深度 | 必做问题 | 选做问题 | 不做的问题 |
|------|---------|---------|-----------|
| 🟢 深入 | Why + How + What if + 并发 + 底层原理 + 数据 | 边界极端场景 | 无 |
| 🟡 基本 | Why + How + What if | 并发 + 数据 | 底层原理（标注⚠️） |
| 🔴 仅会用 | Why + How | What if（简化） | 底层原理 + 并发 + 数据 |

### 问题类型清单

| 类型 | 示例 | 优先级 |
|------|------|--------|
| Why 不选 X | "为什么不用 Kafka 而用 RabbitMQ？" | 必做 |
| How 实现 | "分片上传的具体流程是什么？" | 必做 |
| What if 故障 | "Redis 挂了怎么办？" | 必做 |
| 并发/竞态 | "两个线程同时执行会怎样？" | 选做 |
| 底层原理 | "singleflight 的实现原理？" | 🟢必做 / 🟡选做 / 🔴不做 |
| 性能数据 | "你的 QPS 数据怎么测的？" | 有数据就必做 |
| 边界极端 | "上传到99%断网了怎么办？" | 选做 |

## 第五层：兜底 / 故障场景

### 目的
展示系统设计思维——任何系统都不是完美的，重要的是知道如何应对故障。

### 关键：回答要具体到代码层面

❌ "加个重试就行了" → 太泛
✅ "代码中捕获了 MQ 发送异常，在 catch 块中直接走 `applyLike` 同步写库。同步写的事务包含 INSERT + UPDATE，大约 5-10ms"

### 模板
```
### 兜底保障

#### Q1: [环节A] 失败了怎么办？
[具体的兜底方案：哪段代码处理了、用户看到什么、数据最终状态是什么]

#### Q2: [环节B] 挂了怎么办？
[同上]
```

### 通用兜底模式速查

| 故障类型 | 兜底策略 | 适用场景 |
|----------|---------|---------|
| 消息发送失败 | ACK 确认 + 重试（指数退避） | MQ 不可达 |
| 重复消费 | 幂等设计（唯一索引 / 分布式锁 / 去重表） | MQ 重复投递 |
| 服务宕机 | 降级（切换备用方案）+ 限流 + 熔断 | 核心依赖不可用 |
| 数据不一致 | 最终一致性（Outbox / 重试 / 补偿 / 对账） | 分布式双写 |
| 缓存击穿 | 互斥锁 + 逻辑过期 + singleflight | 热点 Key 失效 |
| 缓存穿透 | 布隆过滤器 + 缓存空值 | 恶意查询不存在的数据 |
| 缓存雪崩 | 随机过期时间 + 多级缓存 | 大量 Key 同时失效 |
| 资源耗尽 | 限流（令牌桶/固定窗口）+ 削峰（MQ）| 流量突增 |
| 第三方 API 失败 | 重试 + 降级 + 死信队列 | 网络抖动/限流 |
