③ 业务Service 编排核心逻辑
④ 持久化Repository 写库 + 缓存
⑤ 响应统一封装返回 JSON
关键设计

Part 2 / 设计

三个最值得借鉴的设计决策

01

分层清晰

Handler / Service / Repository 严格分层，依赖单向向下，便于测试与替换。

02

配置外置

所有环境差异通过配置注入，代码与环境解耦，一份镜像跑遍所有环境。

03

可观测

结构化日志 + 链路追踪，线上问题能快速定位到具体请求与代码位置。

Part 2 / 目录结构

关键目录与各自职责

路径职责
internal/handler/HTTP 入口，参数绑定与响应封装
internal/service/业务逻辑编排
internal/repo/数据访问层
pkg/middleware/鉴权、日志、限流中间件
入口很薄Handler 只做 IO 转换，不含业务。
业务集中所有规则收敛到 Service 层。
依赖倒置Service 依赖接口，不依赖具体实现。
