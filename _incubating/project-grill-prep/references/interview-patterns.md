# 常见面试追问模式

本文档整理了项目面试中最常见的追问模式。

---

## 一、Why 系列（为什么）

这类问题考察技术判断力——你是否基于业务场景做出了合理选择。

**回答框架（四步法）：**
1. 先承认替代方案的合理性（"X 确实有 Y 的优势"）
2. 点出替代方案在**当前业务场景**下的致命缺陷
3. 说明所选方案如何恰好解决这个缺陷
4. 补充务实考量（成本、技术栈、运维）

| 追问模式 | 示例 |
|----------|------|
| 为什么选 X 不选 Y | "为什么用 RabbitMQ 不用 Kafka？" |
| 为什么不采取更简单的方案 | "为什么不直接用线程池？" |
| 为什么这么设计 | "为什么把合并的决策交给前端？" |
| 为什么用这个数据结构 | "为什么用 ZSET 不用普通 SET？" |

---

## 二、How 系列（怎么做的）

这类问题考察你是否真正做过——纸上谈兵的人会在细节上露馅。

**回答框架：**
1. 按时间线描述完整流程（先做什么，再做什么）
2. 重点讲关键分支（if...else...的判断条件和处理方式）
3. 给出具体的数据结构示例（Redis Key 长什么样）
4. 性能数据要说明测试方法（JMeter / Go benchmark / 压测脚本在哪）

| 追问模式 | 示例 |
|----------|------|
| 具体实现流程 | "分片上传的具体流程是什么？" |
| 数据结构设计 | "Redis 里存的什么？Key 怎么设计的？" |
| 关键代码逻辑 | "合并的时候怎么保证幂等？" |
| 性能数据怎么测 | "你的 8 倍提升怎么测的？" |

---

## 三、What if 系列（如果XX怎么办）

这类问题考察系统设计思维和容错意识——高级开发和初级开发的核心区别。

### 回答框架
1. 不要慌，这类问题没有完美答案，考察的是思路
2. 先定性：这是哪类问题（并发 / 故障 / 极端场景）
3. 给出具体方案：预防（怎么做）+ 检测（怎么发现）+ 恢复（怎么修）
4. 说明方案的代价和适用场景

### 3.1 故障场景

| 追问模式 | 兜底策略 |
|----------|---------|
| MQ 不可用 | 降级为同步写库（代码中要有 fallback 路径） |
| 重复消费/提交 | 幂等设计（唯一索引 / OnConflict DoNothing / 去重表） |
| 服务宕机 | 降级 + 限流 + 熔断 |
| 缓存失效 | 互斥锁 / singleflight / 逻辑过期 |

### 3.2 并发场景

| 追问模式 | 示例 |
|----------|------|
| 竞态条件 | "最后几个分片同时完成，同时触发合并怎么办？" |
| 超卖/重复 | "如何保证同一用户只能点赞一次？" |

### 3.3 极端场景

| 追问模式 | 示例 |
|----------|------|
| 大数据量 | "千万级视频怎么排热榜？" |
| 网络中断 | "上传到99%断网了怎么办？" |
| 恶意请求 | "有人疯狂刷接口怎么办？" |

## 四、底层原理系列

这类问题考察技术深度。**注意：只在深度等级为 🟢 时主动准备这类问题。**

| 技术领域 | 常见追问 |
|----------|---------|
| Redis | Lua 脚本为什么能保证原子性？持久化机制？内存淘汰策略？ |
| 消息队列 | 消息投递保证？消费模式？死信队列？ |
| 分布式锁 | WatchDog 原理？Redisson vs SETNX？锁的可重入？ |
| 数据库 | 索引为什么用 B+ 树？事务隔离级别？MVCC？ |
| Go 并发 | channel vs mutex？goroutine 调度？singleflight 原理？ |
| 缓存 | 缓存击穿/穿透/雪崩的区别和解决方案？ |

**保守策略：**
- 只讲自己真正理解的
- 不确定的可以说"我理解到 X 层面，更底层的 Y 机制我还在学习"
- 一句话概括原理，再展开细节

## 五、AI Coding 系列（vibe coding 项目必考）

| 追问 | 考察点 | 推荐回答策略 |
|------|--------|------------|
| "这些不都是 AI 帮你写的吗？" | 是否真正理解代码 | 用具体案例证明：问题是你识别的，方案是你从 AI 的建议中选择的，质量是你把关的 |
| "AI Coding 的具体流程？" | 是否有方法论 | 需求拆解 → 选型验证 → 分模块实现 → review → debug，每步举具体例子 |
| "AI 的不足之处？" | 批判性思维 | 幻觉（AI 编造不存在的 API）、长上下文遗忘、调试能力有限 |
| "0→1 用 AI 解决问题的案例" | 能否举例 | 准备一个具体的并发 bug 案例：发现问题 → AI 给方案 → 你选方案 → 你验证 |
| "更前沿的 AI Coding 方式？" | 技术视野 | 提到 Agentic Coding、Multi-Agent、MCP、Context Engineering |

**核心话术：** "AI 是高级搜索引擎和代码助手，不是项目经理。需求理解、架构决策、质量把关始终是我自己做的。"

## 六、项目延展系列

| 追问 | 考察点 | 回答策略 |
|------|--------|---------|
| "如果用户量增长10倍？" | 扩展性思维 | 分短期/长期给方案，展示成本意识 |
| "这个项目的局限？" | 自我认知 | 先承认局限比说"没有局限"好得多 |
| "你会怎么改进？" | 产品思维 | 具体方案 + 分优先级 |
| "如果重新设计？" | 反思能力 | 说1-2个"当时没想到但现在会改"的点 |

---

## 七、"不该说什么"判断指南

在面试中，以下情况应该避免主动提及：

| 情况 | 为什么危险 | 替代策略 |
|------|-----------|---------|
| 你用了但完全不理解原理的技术 | 面试官一追问就暴露 | 不主动提，被问到说"这里我主要关注的是使用场景" |
| 你实现过但有已知 bug 的功能 | 等于主动暴露弱点 | 被问到时说"这部分我还在优化" |
| 纯粹为了用而用的技术 | 面试官一眼就能看穿 | 不提，或者在文档中标注为"选型争议点"准备好解释 |
| 你编造的量化数据 | 面试官会追问测试方法 | 不编数据，说"这是我根据压测估算的"或直接说没测过 |

## 八、深度探测问题库

以下问题用于阶段二的深度探测，每个技术领域 1-2 个快速诊断题。

### 缓存类

| 诊断问题 | 🟢 深入 | 🟡 基本 | 🔴 仅会用 |
|---------|--------|--------|----------|
| "缓存回源时100个并发同时miss，你的系统怎么处理的？" | 讲出 singleflight 合并或分布式锁串行化的具体机制 | 知道要防缓存击穿但说不出具体实现 | 不知道这个问题 |
| "L1缓存3秒过期和Redis缓存5分钟过期，这两个TTL是怎么定的？" | 能说出基于访问频率分析和压测调参 | 知道是经验值但说不清为什么 | 不知道 |

### 消息队列类

| 诊断问题 | 🟢 深入 | 🟡 基本 | 🔴 仅会用 |
|---------|--------|--------|----------|
| "为什么不在事务里直接发MQ，而要用Outbox表？" | 能说出"事务提交前宕机丢消息"和"本地事务保证+轮询投递"的完整逻辑 | 知道"保证一致性"但说不出具体场景 | 不知道为什么 |
| "MQ消费失败后你的重试策略是什么？最多重试几次？" | 能讲出 Nack + requeue + DLX + max retry 的完整链路 | 知道有重试但说不出具体次数和兜底 | 不知道 |

### 幂等/并发类

| 诊断问题 | 🟢 深入 | 🟡 基本 | 🔴 仅会用 |
|---------|--------|--------|----------|
| "MQ重复投递了一条点赞消息，你的系统怎么处理的？" | 讲出唯一索引 + OnConflict DoNothing 的代码级实现 | 知道要幂等但说不出代码层面的实现 | 不知道什么是幂等 |
| "两个用户同时对同一个视频点赞，你的likes_count会错吗？" | 能分析出 UPDATE likes_count + 1 是行级锁，不会错 | 知道不会有问题但说不清为什么 | 不确定 |

### 数据结构/算法类

| 诊断问题 | 🟢 深入 | 🟡 基本 | 🔴 仅会用 |
|---------|--------|--------|----------|
| "你的分钟级ZSET是怎么聚合60个窗口的？用户翻页时怎么保证不重复？" | 讲出 ZUNIONSTORE 聚合 + as_of 快照时间戳 + offset 分页 | 知道用 ZSET 但说不出聚合和分页细节 | 不清楚 |
| "你的热点Key检测器的滑动窗口是怎么实现的？" | 讲出分桶计数 + 周期轮转的 ring buffer 思想 | 知道是滑动窗口但说不出具体实现 | 不清楚 |

### 系统设计类

| 诊断问题 | 🟢 深入 | 🟡 基本 | 🔴 仅会用 |
|---------|--------|--------|----------|
| "你的限流方案在高并发下有什么缺陷？" | 能讲出固定窗口的边界突刺问题，以及令牌桶/滑动窗口的改进方案 | 知道有限流但说不出具体方案的局限 | 不清楚 |
| "如果Redis完全不可用，你的Feed接口还能用吗？" | 能讲出 L1 本地缓存 + 直连 DB 的降级路径，以及性能退化的预期 | 知道要降级但说不出具体路径 | 不知道 |
