# Skill: interview-scenario-answer

# 面试场景题回答指南

Iron Law: **永远先说思考框架，再讲具体方案。** 面试官考察的是你的思维方式，不是背答案的能力。展示"我是怎么想的"比"答案是什么"更重要。

## 触发场景

用户问以下类型的问题时触发：
- 面试题/场景题："这道面试题怎么回答"
- 系统设计题："面试官问xxx怎么设计"
- 边界场景题："如果xxx出了问题怎么办"
- 项目相关题："这个功能模块怎么讲"
- 八股延伸题："xxx原理是什么"

触发关键词：面试、怎么答、怎么回答、面试官问、场景题、系统设计、边界、八股、原理

## 工作流程

### Step 1: 理解题目 ⚠️ REQUIRED

分析面试题的类型和考察意图：
- **系统设计题**: 考察架构能力、技术选型、权衡取舍
- **场景题**: 考察边界处理、异常处理、兜底策略
- **原理题**: 考察技术深度、八股掌握程度
- **项目题**: 考察项目理解、技术亮点、解决问题能力

### Step 2: 构建思考框架 ⚠️ REQUIRED

**问题拆解公式**:
```
1. 这个问题的核心是什么？（本质）
2. 面试官想听什么？（意图）
3. 我应该怎么组织回答？（结构）
```

**回答顺序原则**:
- 先说**思考框架**（展示思维方式）
- 再说**通用方案**（展示技术广度）
- 然后**结合项目**（展示实践经验）
- 最后**延伸八股**（展示技术深度）

### Step 3: 设计通用方案 ⚠️ REQUIRED

**技术选型公式**:
```
我选择 X 技术，因为：
1. 它能解决 Y 问题
2. 相比 A 方案，它的优势是...
3. 相比 B 方案，它的劣势是...
4. 在当前场景下，它是最佳选择
```

**边界处理清单**:
- 输入边界：空值、null、超长、格式错误
- 并发边界：竞态条件、死锁、资源争抢
- 时间边界：超时、重试、幂等
- 资源边界：内存溢出、磁盘满、连接池耗尽
- 网络边界：断网、弱网、DNS解析失败

### Step 4: 结合项目（如果有） ⚠️ REQUIRED

**项目结合公式**:
```
在我负责的 XXX 项目中，我们遇到了类似的问题：
1. 业务背景：...
2. 技术方案：...
3. 实现细节：...
4. 最终效果：...
```

**亮点提炼**:
- 性能提升：p90延迟降低 XX%
- 稳定性提升：故障率降低 XX%
- 成本优化：资源占用减少 XX%
- 用户体验：响应时间降低 XX%

### Step 5: 准备面试话术 ⚠️ REQUIRED

**30秒版本**（快速回答）:

## 写作原则

### 展示思维方式，不是背答案

**反模式**:
```
答案是：使用 Redis 缓存。
```

**正确方式**:
```
思考框架：
1. 首先分析问题本质：这是一个高频读、低频写的场景
2. 然后考虑技术选型：对比 Redis、Memcached、本地缓存
3. 最后结合场景：选择 Redis，因为需要持久化和丰富数据结构
```

### 对比思维，展示权衡能力

**反模式**:
```
使用方案 A。
```

**正确方式**:
```
对比三个方案：
- 方案 A：优点是...，缺点是...
- 方案 B：优点是...，缺点是...
- 方案 C：优点是...，缺点是...

在当前场景下，选择 A，因为...
```

### 边界处理，展示全面性

**反模式**:
```
正常情况下这样做。
```

**正确方式**:
```
正常流程：...
边界处理：
- 如果输入为空：...
- 如果超时：...
- 如果并发冲突：...
- 如果资源不足：...
```

```
## 通用方案设计

### 业务背景
**场景**: 用户需要上传大文件（视频、数据集、备份文件）
**痛点**: 网络不稳定、上传时间长、失败需要重传
**目标**: 支持断点续传、并发上传、失败重试

### 技术选型

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 前端分片+后端合并 | 灵活、可控 | 实现复杂 | 大文件上传 |
| 客户端直传OSS | 节省服务器 | 不可控 | 简单场景 |
| 秒传+分片 | 体验最好 | 需要MD5计算 | 重复文件多 |

**选择**: 前端分片+后端合并，因为需要控制上传流程和权限

### 核心实现

**分片策略**:
- 分片大小：5-10MB（太小请求多，太大重传成本高）
- 4GB / 5MB = 819个分片
- 前端并发3-5个，避免浏览器连接数限制

**状态管理**:
- 方案一：Redis Bitmap（推荐，空间效率高）
  - 每个bit表示一个分片状态
  - 1000分片只需125字节
- 方案二：Redis Set（简单，但空间占用大）
- 方案三：数据库查询（持久化，但性能差）

**断点续传流程**:
1. 前端计算文件MD5（分片计算，不加载全文件）
2. 调用后端接口查询已上传分片
3. 只上传缺失的分片
4. 上传完成后调用合并接口

### 边界处理与兜底策略

**幂等性处理**:
```
问题: 服务端超时但分片已上传，客户端重传
方案: 服务端幂等处理
1. 唯一标识: fileHash + chunkIndex
2. 上传前检查: if exists -> return success
3. 原子操作: Redis SETNX 或 对象存储覆盖写
```

**客户端crash恢复**:
```
方案: 双层状态持久化
1. 客户端: localStorage记录上传进度
2. 服务端: Redis/DB记录已接收分片
3. 恢复时: 查询服务端状态，只传缺失分片
```

**弱网环境**:
```
方案: 重试+指数退避
1. 超时阈值: 根据分片大小调整（5MB建议30s）
2. 重试策略: 指数退避 1s, 2s, 4s, 8s
3. 最大重试: 3-5次
```

### 延伸八股

**Q: 为什么用Redis Bitmap不用Set？**
A: Bitmap用1个bit表示一个分片，1000分片只需125字节；Set每个元素至少几十字节。Bitmap查询O(1)，空间效率提升100倍+。

**Q: 如何保证分片上传的幂等性？**
A: 用fileHash+chunkIndex作为唯一键，上传前检查是否已存在。如果存在直接返回成功，不存在才执行上传。

**Q: 前端如何计算大文件的MD5？**
A: 分片计算，不加载全文件到内存。将文件分成多个chunk，逐个计算MD5，最后拼接计算整体MD5。
```

```
### 追问准备
**Q: 如果并发量增加10倍，怎么优化？**
A: 前端限制并发数（如3-5个），后端用消息队列异步处理合并，数据库批量写入分片记录。

**Q: 如果这个服务挂了，怎么办？**
A: 分片状态持久化到Redis，服务重启后可恢复；合并操作支持幂等，可重试。

**Q: 如果让你重新设计，你会怎么改进？**
A: 增加秒传功能，计算文件MD5直接返回已有文件；增加直传OSS，减轻服务器压力。
```

```
## 通用方案设计

### 业务背景
**场景**: 企业知识库管理系统，用户上传文档后可进行智能问答
**痛点**: 文档格式多样、处理耗时、需要向量化支持语义搜索
**目标**: 支持多格式文档上传、异步处理、向量化存储

### 整体架构

```
用户上传 → 分片存储 → Kafka异步 → 文档解析 → 分块 → 向量化 → ES存储
    ↓           ↓           ↓           ↓         ↓         ↓
  前端并发    MinIO       解耦       Tika      语义分块   Embedding
```

### 核心模块

**1. 分片上传模块**
- 技术选型：前端分片 + MinIO存储
- 状态管理：Redis Bitmap
- 断点续传：查询已上传分片，只传缺失部分

**2. 异步处理模块**
- 技术选型：Kafka
- 优势：上传秒返、解耦处理、支持重试
- 流程：上传完成 → 发送Kafka消息 → 消费者处理

**3. 文档解析模块**
- 技术选型：Apache Tika
- 支持格式：PDF、Word、Excel、PPT等20+格式
- 流式处理：避免大文件OOM

**4. 文本分块模块**
- 策略：父文档-子切片二级切分
- 语义保持：段落 → 句子 → HanLP词语
- 目的：保持切片语义完整性

**5. 向量化模块**
- 技术选型：Embedding API
- 批量处理：一次API调用处理所有文本块
- 存储：Elasticsearch + 向量字段

### 技术亮点

1. **异步解耦**: Kafka实现上传和处理分离，上传秒返
2. **流式解析**: Tika流式处理，避免大文件OOM
3. **语义分块**: 三级切分保持语义完整性
4. **批量向量化**: 一次API调用处理所有文本块
5. **权限携带**: 向量文档携带权限信息，支持检索时过滤

### 边界处理

**大文件OOM**:
- 流式解析，不加载全文件到内存
- 内存阈值检查，超过80%触发GC

**解析失败**:
- Kafka重试机制
- 死信队列处理多次失败

**向量化失败**:
- 抛出异常，Kafka重试
- 记录日志，人工介入

### 延伸八股

**Q: 为什么用Kafka不用RabbitMQ？**
A: Kafka吞吐量高，适合大文件处理场景；支持消息持久化，可回溯；支持消费者组，可水平扩展。

**Q: 父文档-子切片策略的优势？**
A: 父块保留完整上下文，用于生成时提供背景；子块精确匹配查询，用于检索。兼顾精度和完整性。

**Q: 如何保证向量化的一致性？**
A: 向量化失败抛出异常，Kafka重试；向量文档包含fileMd5和chunkId，支持幂等写入。
```
