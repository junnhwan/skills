# 样本 3：知广知识分享平台

> 来源：用户提供的简历截图（D:\Download\3.png）
> 时间段：未标注

## 各子系统

- **认证系统**：开发基于 Spring Security 的 JWT 双令牌认证系统，采用 RS256 签名 + Redis 刷新令牌白名单，实现 15 分钟访问令牌 + 7 天刷新令牌的安全会话管理，支持即时令牌撤销，兼顾安全与高性能。

- **计数系统**：笔记维度（点赞收藏）与用户维度（关注取关）以 Redis 作为底层存储系统，采用定制化 Redis SDS 二进制紧凑计数，使用 Lua 脚本进行原子更新，并实现了采样一致性校验与自愈重建。

- **发布系统**：采用渐进式发布流程，发布的图片、视频、Markdown 文档等都存入 OSS 对象存储系统，采用后端发布预签名 + 前端直传的形式上传，节省前后端传输资源。并接入 DeepSeek AI 一键生成文章摘要。

- **点赞系统**：采用异步写 + 写聚合 Kafka 异步写 + 写聚合的形式应对高并发写场景。采用位图的结构高效实现幂等和判重。读取遇到异常或缺失时，基于位图做按需重建，保证最终一致。并用 Kafka 做"灾难回放"的兜底操作。

- **用户关系系统**：实现关注功能，采用一主多从 + 事件驱动模型。粉丝表、计数系统、列表缓存都作为关注系统的伪从。关注事件发生时，在同一事务中插入关注表和 Outbox 表，使用 Canal 订阅 Outbox 表的 binlog，并将变更事件发布到 Kafka 异步更新其他数据源。

- **Feed 流**：采用三级缓存架构且设计了缓存一致性策略，本地 Caffeine + Redis 页面缓存 + Redis 片段缓存。自定义 hotkey 探测机制基于热点检测按层级延伸缓存时长，叠加随机抖动抗雪崩。并设置单飞锁（single-flight）避免同一页并发回源风暴。

## 这份样本好在哪

- **没有项目简介一句话先讲清楚就直接列子系统** —— 简历里的"项目"不是论文，子系统名 + 一段话即可，读者已经知道这是个内容平台。

- **每条子系统的写法骨架完全一致**：
  ```
  X 系统：采用/使用/基于 Y 实现 Z，[具体技术细节]，[兜底/容错/性能策略]。
  ```
  读者一眼就能扫过去找自己关心的那一块。

- **每个名词都为某个具体痛点服务**：
  - 位图（bitmap）→ 幂等和判重 + 异常时按需重建
  - Outbox + Canal + Kafka → 关注事件跨数据源最终一致
  - 单飞锁（single-flight）→ 避免并发回源风暴
  - hotkey 探测 → 热点 key 缓存时长延伸 + 随机抖动抗雪崩

  没有一个名词是为了"显得技术深"出现的，每个都跟一个具体场景绑定。

- **不报"我提升了多少"**：这份没有任何"+X%"或"从 X 降至 Y"的数字，但读起来仍然硬。原因是每个技术决策都有清晰的"为什么用它"，工程师看了知道"这人真做过"。
  → **教训**：没数字也能写硬话，前提是每个名词都解释清楚为什么用。
