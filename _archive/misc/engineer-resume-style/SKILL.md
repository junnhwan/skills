# Engineer Resume Style

## 核心理念

工程师对工程师讲话。简历是工程文档，不是访谈稿，不是 PPT，不是论文 abstract。

三条铁律：

1. **每个名词都为某个具体痛点服务**。"PGVector"出现是因为它解决了"文本向量化存储"，不是因为它新潮。读者扫到一个技术名词时，应该立刻能在同一句话里看到它解决的问题。
2. **每个数字都可以被面试官追问**。"耗时 15s 降至 3s"、"召回率 85%"、"首字延迟 < 1 秒"——每个数字都对应一个可重现的测量场景。"显著提升 / 大幅优化"是修辞不是数字。
3. **简历里 0 个"对标"是健康标准**。"对标 LangGraph"是自报家门式的不专业。让面试官自己问"你怎么想到用这个的"，他来问时再讲来源。

## 何时触发（pushy 触发清单）

主动触发场景：

- 用户说"写简历"、"项目段落"、"项目经历"、"技术亮点"
- 用户说"重写这段"、"改写"、"优化简历"、"polish 一下"
- 用户抱怨现有简历"太浮夸"、"太装"、"堆名词"、"太虚"、"假大空"
- 用户给参考样本说"参考别人这样写"、"换个风格"
- 用户说"硬一点"、"工程化一点"、"不要那么飘"
- 用户在面试 / 求职上下文（讨论 JD、备战面试）下贴出现有简历

非触发：英文简历（这个 skill 是中文专用）、非工程领域简历（产品、设计、运营等）。

## 写作骨架

### 项目段落整体结构

```
## 项目名 | 角色 | 时间段

**项目简介**: 一两句业务价值角度的描述（做什么 + 给谁用 + 核心能力）。

**技术栈**: A + B + C + D + E（一行平铺，无修饰）

### 核心亮点

- **主题 1**: [核心做法]，[具体方案]，[数字或兜底]
- **主题 2**: [核心做法]，[具体方案]，[数字或兜底]
- **主题 3**: [核心做法]，[具体方案]，[数字或兜底]
- **主题 4**: [核心做法]，[具体方案]，[数字或兜底]   (可选)
```

3-4 个亮点是简历项目段落的甜蜜区。少于 3 撑不起项目分量，多于 4 信息密度稀释、读者注意力涣散。

### 单个亮点的结构

每个亮点是一个 **(主题，主路径，量化或兜底)** 三元组：

```
**主题**: 基于/采用/使用 X 实现 Y，[具体方案细节]，[可量化结果 或 兜底/降级/容错策略]
```

变体：

- **主题**: 用 X 替换 Y，解决 Z 问题，[数字]
- **主题**: 设计 X 架构（A + B + C 三层），[每层做了什么]，[整体效果]
- **主题**: 集成 X + Y + Z，[组合在一起达成什么]，[兜底策略]

主题词建议用**功能领域**而不是**技术名**：
- ✅ "异步入库管线" / "权限与文件处理" / "流式对话与降级容灾"
- ❌ "Kafka 模块" / "Spring Security 模块"（直接用技术名作主题，读者要先反推这个技术解决什么问题，多一道弯）

## 禁忌词清单（直接全文搜删）

### 1. 修辞型形容词

| 禁忌词 | 为什么不要 | 替代写法 |
|---|---|---|
| 现代化 / 工程化 / 标准化 | 不指代任何具体动作。如果工作是"工程化的"，讲具体做了什么让它工程化 | 删掉，描述具体动作 |
| 打磨 / 锤炼 / 沉淀 / 精细化 | 比喻型动词，不是工程动词 | 改用"实现 / 构建 / 重构 / 设计" |
| 全链路 / 端到端 / 一体化 | 主审人不会被这种词打动，要看你做了哪几个环节 | 列具体环节："上传 → 解析 → 入库 → 检索" |
| 智能化 / 自动化 / 高性能 | 形容词，必须改成具体能力 | "自动化"→ "自动 X"（X 是具体动作） |
| 深度 X / 完整 X / 极致 X | 修辞型 modifier | 删掉 modifier，留 X |

### 2. 结构化吹嘘

| 禁忌词 | 为什么不要 |
|---|---|
| 三大核心能力 / 五大模块 / X 大维度 | "三大"是 PPT 用语。简历里直接列三个名字更工程化（"知识库、对话、运维"） |
| 闭环 / 全闭环 / 形成闭环 | 这是产品经理的词。工程师写"反馈环 / 迭代到 X 收敛"更具体 |
| 体系 / 生态 / 矩阵 | 同上，PPT 抽象词 |

### 3. 对标病

| 禁忌词 | 为什么不要 |
|---|---|
| 对标 LangGraph / Anthropic / OpenAI / xx 论文 | 简历里写"对标 X"是自报家门式的不专业。技术决策来源应该让面试官自己问出来 |
| 对齐业界标准 / 业界领先实现 | 同上，更虚 |
| 借鉴 X 思路 | 删掉"借鉴"，直接讲做了什么 |

### 4. 故事化与第一人称

| 禁忌词 | 为什么不要 |
|---|---|
| 三幕剧 / 起承转合 / root cause 故事 | 这是面试 STAR 答案不是简历写法。简历是工程文档不是访谈稿 |
| 我意识到 / 我注意到 / 我发现 | 第一人称叙事不进简历。简历用动词无主语（"实现 / 构建 / 重构"） |
| 拍出 / 揪出 / 抓出 silent bug | 修辞 + 故事化，双重违规 |

### 5. 虚数字

| 禁忌词 | 为什么不要 |
|---|---|
| 显著提升 / 大幅优化 / 有效降低 | 没有数字的"提升"等于没说 |
| 提高 30%（不带分母） | 必须带"在什么数据集 / 什么场景下"的分母 |
| 性能优秀 / 性能领先 | 形容词不是数字 |

## 数字哲学

### 三种合格的数字

1. **绝对数字 + 量纲**：QPS、耗时（ms/s/min）、文件大小（MB/GB）、行数、case 数
   - "首字延迟 < 1 秒"
   - "1GB 文件上传耗时 15s 降至 3s"
   - "116 JUnit 测试全绿"

2. **比例**：X 倍 / X% / N×M → N+M（这是协议化成果）
   - "效率提升 92%"
   - "工具集成成本从 N×M 降到 N+M"
   - "处理效率提升 3 倍"

3. **量级跨越**：小时 → 分钟、分钟 → 秒（精度低但仍是硬话）
   - "告警响应时间从小时级降低至分钟级"
   - "故障分析从 40 分钟降至 3 分钟"

### 没数字时的两种合格写法

不是每个项目都有完整的 benchmark。没数字时仍能写出硬话，靠这两条：

1. **架构特点描述**：讲清楚"用了什么 + 解决什么 + 怎么实现"。
   > 例：producer 事务 + idempotent + DLT 兜底，consumer 用 ES 主键 chunkId 幂等吃掉 at-least-once 重投递

   没有数字但工程师看了立刻知道你做了什么、解决什么消息可靠性问题。

2. **兜底/容错策略**：讲清楚"失败怎么办"。
   > 例：Embedding 调用失败时自动降级到关键词检索保障可用性

   没数字但显示工程素养。

参考 `references/sample-3-zhiguang.md` —— 全文 0 个"+X%"数字，仍然是硬简历。

### 待测数字的占位规则

用户的项目还在迭代，benchmark 没跑完是常态。**永远不要凭空编数字**。用占位标注待填：

```
"1GB 文件上传 [TBD: 耗时 X 秒]"
"[TBD: v3 hybrid 在 holdout 集上 Recall@5 比 v2 dense-only 提升 X]"
"消费 latency [TBD: 平均 X 秒, p95 Y 秒]"
```

写简历时把所有 TBD 集中标在文末"待测清单"里方便用户后续回填。

## 写作流程

1. **先把现有素材列清楚**（与用户确认）
   - 技术栈（用了什么 —— 一行能列完的清单）
   - 已知硬数字（有数字的就用）
   - 待测数字（有可能测但还没跑的，用 TBD 占位）
   - 项目时间段、角色（独立 / 团队 / 实习）

2. **决定项目分割粒度**
   - 一个大项目跨越多个阶段 → 倾向单项目 + 3-4 个亮点（每阶段一个亮点）
   - 几个独立交付物 → 倾向多项目，每个项目内 3 个亮点
   - 简历空间紧（< 半页）→ 合并成单项目
   - 简历空间松（半页 +）→ 拆开能讲更多技术点

3. **草稿先按骨架填**
   - 项目名 + 时间 + 角色（一行）
   - 项目简介（业务角度 1-2 句）
   - 技术栈（一行）
   - 亮点 1~4（每条 2-3 句完整 (主题，主路径，量化或兜底) 三元组）

4. **过禁忌词清单**
   - 全文搜：现代化、工程化、闭环、对标、X 大、深度、完整、全链路、显著、大幅
   - 找到就删或重写

5. **检查每个名词的目的性**
   - 逐条读亮点：每个出现的技术名词是为了解决什么具体问题？
   - 解释不出来的删掉
   - 解释得出但读起来突兀的，重新组织顺序

6. **最后扫一遍数字**
   - 每个数字带分母 / 量纲了吗？
   - 没数字的地方有没有用"架构特点 / 兜底策略"补强？

## 一段示例：同一事实，反例 vs 改造后

### 反例（修辞 + 对标 + 故事）

> **【RAG 升级三件套】** `KnowledgeStoreIds.computeChunkId` 用 sha256(content) 替代自增 int 实现幂等 ingest;`ContextualChunkAnnotator` 用 LLM 给每 chunk 生成上下文前缀 → embed `prefix\n\nchunk`(metadata.rawText 仍存原文供 evidence/lexical 使用);`RrfFusionMerger` 用 RRF(k=60)替代手写线性加权 + bonus,跨模式无需分数校准。**对标 Anthropic Contextual Retrieval / Vespa & Elastic 标准做法**

问题：
- "三件套" 是结构化吹嘘
- "对标 Anthropic / Vespa & Elastic" 是对标病
- 信息密度堆得过高，读者一口气吃不下

### 改造后

> **RAG 入库与检索**: 用 sha256(content) 生成 chunkId 实现幂等入库，重复上传不会污染索引；用 LLM 给每段文档生成上下文前缀后再 embed，原文存 metadata 供后续 evidence 引用；混合召回用 RRF (k=60) 融合 dense 和 BM25，避免手写线性加权要校准两路分数量级的麻烦。

差别：
- 删除"三件套" → 改成"入库与检索"两个动作分组
- 删除"对标 Anthropic" → 让"上下文前缀"自己说明做法来源
- 每个技术点跟一个具体痛点绑定（污染索引 / evidence 引用 / 校准分数）
- 没数字但每个名词都有目的，读起来仍然硬

## 输出形式约定

- 默认输出 Markdown
- 项目段落用 `##` 二级标题
- 亮点用 `-` 加粗主题前缀的列表
- 待测数字一律 `[TBD: 描述]` 占位，文末汇总一份"待测清单"方便用户回填
- 不主动加 emoji（用户明示要才加）
- 不主动写 "面试一句话讲法 / electronic pitch / 三幕剧叙事" —— 这些是面试材料不是简历内容

## 与用户协作的关键检查点

写简历草稿前必须先问清楚（如果上下文里没有）：

1. 简历空间是 1/2 页还是 1 整页？决定项目分割粒度。
2. 角色是独立项目还是团队项目？决定主语用法。
3. 已知硬数字有哪些？没列全的话不要瞎编。
4. 项目要写几个独立段落（一个大项目 vs 拆成多个）？

写完草稿后主动提示用户：

- 哪些位置标了 [TBD]，需要后续测什么数字
- 哪些技术名词还没写清楚"为什么用它"，需要用户补充上下文
- 删掉了哪些原稿里的内容（如果是重写）以及为什么删
