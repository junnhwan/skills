# Interview Scenario Answer

## 核心理念

**先讲思考框架，再讲具体方案。** 面试官考察的是你的思维方式，不是背答案的能力。一个能清晰拆解问题、做技术对比、讨论边界的候选人，永远比一个只会报答案的候选人值钱 —— 哪怕答案完全一样。

所以这个 skill 的输出永远是"组织化的回答"，不是"答案"。它需要把题目拆开，让面试官看见你的思考路径。

## 三个核心原则

### 1. 展示思维方式，不是背答案

面试官问"高频读场景怎么做"，反模式是直接说"用 Redis"。正确方式是先说"这是一个高频读、低频写的场景，所以缓存是合理选择 → 然后对比 Redis / Memcached / 本地缓存 → 最后落到 Redis 因为需要持久化和丰富数据结构"。读起来多一倍，但展示了你的思维。

### 2. 对比思维，展示权衡

只说"用方案 A"暴露的是经验单薄。正确方式是列出 A / B / C 三个方案的优缺点和适用场景，然后说"在当前场景下选 A，因为 ..."。哪怕你选的方案是显而易见的，也要让面试官看见你考虑过其他选项。

### 3. 边界处理，展示全面性

只讲正常流程暴露的是工程经验不足。每个回答都要带上：输入边界（空值、超长、格式错误）、并发边界（竞态、死锁）、时间边界（超时、重试、幂等）、资源边界（内存、磁盘、连接池）、网络边界（断网、弱网、DNS）。不是每条都展开讲，但要 demonstrate 你想过这些。

## 工作流程

### 第一步：识别题目类型

不同题型考察的能力不同，组织回答的重点也不同：

| 题型 | 考察重点 | 回答重心 |
|---|---|---|
| 系统设计题 | 架构能力、技术选型、权衡 | 需求 → 架构 → 模块 → 边界 |
| 场景/边界题 | 边界处理、异常处理、兜底 | 正常流程 → 异常场景 → 兜底方案 → 监控 |
| 原理/八股题 | 技术深度、底层理解 | 是什么 → 为什么 → 怎么用 → 对比 |
| 项目题 | 项目理解、解决问题能力 | 业务 → 架构 → 模块 → 亮点 → 踩坑 |

识别错题型会导致回答抓不到重点。先花 5 秒判断题型再组织答案。

### 第二步：构建思考框架

三问拆解：
1. **本质**：这个问题的核心是什么？（剥掉表象）
2. **意图**：面试官想听什么？（猜考点）
3. **结构**：我应该按什么顺序讲？（让对方好跟）

### 第三步：技术选型对比

技术选型公式：
```
我选择 X，因为：
1. 它能解决 Y 问题
2. 相比 A 方案，X 的优势是 ...
3. 相比 B 方案，X 的劣势是 ... （但在当前场景可接受）
4. 综合下来，X 是最佳选择
```

故意提到劣势比假装方案完美更可信 —— 没有银弹是工程师的基本认知。

### 第四步：边界处理清单

回答时按这五个维度过一遍：

- **输入边界**：空值、null、超长、格式错误、特殊字符
- **并发边界**：竞态、死锁、资源争抢、ABA 问题
- **时间边界**：超时、重试策略（指数退避）、幂等保证
- **资源边界**：内存溢出、磁盘满、连接池耗尽、文件句柄
- **网络边界**：断网、弱网、DNS 解析、跨地域延迟

不需要每个都展开，但每条至少提一句"也考虑了 X"。

### 第五步：项目结合（如果有具体项目）

公式：
```
在我负责的 XXX 项目中，遇到过类似的问题：
1. 业务背景：...
2. 技术方案：（呼应前面讲的通用方案）
3. 实现细节：...
4. 最终效果：（带数字）
```

亮点的数字方向：
- 性能：p90 / p99 延迟降低 X%
- 稳定性：故障率 / 错误率降低 X%
- 成本：资源占用 / 调用次数减少 X%
- 体验：响应时间 / 首字延迟降低 X%

数字必须是可被追问的（你必须知道怎么测出来的）—— 不能编。

### 第六步：准备话术的两个版本

**30 秒版本**：核心 + 三点
```
这个问题的核心是 XXX。我会从三个维度回答：
1. 技术选型上 ...
2. 实现方案上 ...
3. 边界处理上 ...
```

**2 分钟版本**：业务背景 → 技术选型 → 实现 → 边界 → 八股
```
面试官您好，这道题我从几个方面来讲：

业务背景上，这个功能解决的是 ...
技术选型上，我对比了 A / B / C，最终选择 X，因为 ...
具体实现上，核心流程是 ...
边界处理上，我考虑了 ...
延伸知识上，相关的原理是 ...
```

### 第七步：追问准备

每道题准备至少 3 个追问的回答。高概率追问：
- "如果并发量增加 10 倍，怎么优化？"
- "如果这个服务挂了，怎么办？"
- "如果让你重新设计，你会怎么改进？"
- "为什么不用 Y 替代 X？"
- "这个方案的瓶颈在哪？"

**回答顺序**: 是什么 → 为什么 → 怎么用 → 对比

## 反模式（明确避开）

- **只给答案不讲思路**："答案是 Redis" → 没思考框架，面试官看不到你的脑子
- **只讲项目没有通用方案**："我们项目这样做" → 面试官以为你只会复读项目
- **只讲实现不讲边界**："正常情况下这样做" → 暴露工程经验不足
- **只讲技术不讲业务**："技术上可以用 X" → 不知道为什么用、什么场景用
- **只背官方文档没有自己的思考**："根据官方文档 ..." → 面试官想听你的判断不是文档摘抄

## 预交付检查清单

回答交付前过一遍：

- [ ] 有明确的思考框架（不是直接给答案）

- [ ] 有边界处理（覆盖至少 3 个维度）
- [ ] 有延伸八股（展示技术深度）
- [ ] 有 30 秒和 2 分钟两个版本
- [ ] 有至少 3 个追问及回答思路
- [ ] 如果用户提到项目，有项目结合（不是空讲理论）

## 示例：4GB 大文件分片上传

**用户输入**：面试官问"4GB 级文件怎么做分片上传和断点续传"，怎么回答？

**输出**：

### 思考框架

**问题本质**：大文件上传的可靠性和用户体验问题
**考察意图**：系统设计能力（分片方案）+ 边界处理（断点续传、异常恢复）+ 技术深度（状态管理、幂等性）
**回答顺序**：整体方案 → 核心实现 → 边界处理 → 延伸八股

### 通用方案设计

**业务背景**：用户上传大文件（视频、数据集、备份），网络不稳定，上传时间长，失败重传成本高。目标是支持断点续传 + 并发上传 + 失败重试。

**技术选型对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 前端分片 + 后端合并 | 灵活、可控、支持权限 | 实现复杂 | 大文件 + 业务控制 |
| 客户端直传 OSS | 节省服务器带宽 | 难做权限校验 | 简单上传 |
| 秒传 + 分片 | 体验最好 | 需要 MD5 计算 | 重复文件多 |

选前端分片 + 后端合并，因为需要业务控制流程和权限。

**核心实现**：

- **分片大小**：5-10 MB（太小请求多，太大重传成本高）。4GB / 5MB = 819 个分片，前端并发 3-5 个避免浏览器连接数限制。
- **状态管理**：用 Redis Bitmap，每个 bit 表示一个分片状态。1000 个分片只需 125 字节，查询 O(1)。对比 Redis Set（每元素几十字节）和数据库查询（性能差），Bitmap 空间效率提升 100 倍+。
- **断点续传流程**：前端分片计算 MD5（流式，不加载全文件） → 调后端查已上传分片 → 只传缺失分片 → 全部完成调合并接口。

**边界处理**：

- **幂等性**：用 fileHash + chunkIndex 作唯一键，上传前检查是否存在。Redis SETNX 或对象存储覆盖写保证原子。
- **客户端 crash 恢复**：双层持久化 —— 客户端 localStorage 记录进度，服务端 Redis 记录已接收分片。恢复时以服务端为准。
- **弱网环境**：超时阈值按分片大小调（5MB 建议 30s），重试用指数退避 1s/2s/4s/8s，最大 3-5 次。

**延伸八股**：

- *为什么 Bitmap 不用 Set？* Bitmap 1 bit / 分片，1000 分片仅 125 字节；Set 每元素几十字节。Bitmap 查询 O(1)，空间效率提升 100 倍+。
- *如何保证分片上传幂等？* fileHash + chunkIndex 作唯一键，上传前 check exists。
- *前端怎么算大文件 MD5？* 分片流式计算，避免加载全文件 OOM。

### 面试话术

**30 秒版本**：

> 4GB 文件按 5MB 分片得到 819 个分片，前端并发 3-5 个上传，每个分片带 fileHash + chunkIndex 作唯一 key。客户端和服务端都记录已上传分片，断点续传时客户端查询服务端状态、只传缺失分片。服务端通过幂等检查保证重传不会重复处理。

**2 分钟版本**：

> 这道题我从几个方面回答。
>
> 业务背景上，大文件上传面临网络不稳定和耗时长的问题，需要断点续传和并发来提升体验。
>
> 技术选型上，我对比了前端分片 + 后端合并、客户端直传 OSS、秒传 + 分片三个方案，选第一个，因为需要业务控制流程和权限。分片大小选 5MB，平衡请求次数和重传成本。
>
> 核心是状态管理。我推荐 Redis Bitmap，每个 bit 表示一个分片，1000 分片只占 125 字节，查询 O(1)。前端用文件 MD5 作唯一 ID 查服务端已上传分片，只传缺失部分。
>
> 边界处理我考虑了三点：幂等性用 fileHash + chunkIndex 作唯一键；客户端 crash 用双层持久化；弱网用指数退避重试。
>
> 延伸知识是 Redis Bitmap 原理和分布式幂等设计。

**追问准备**：

- *并发量增加 10 倍怎么优化？* 前端限并发 3-5 个，后端用 MQ 异步合并，DB 批量写分片记录。
- *服务挂了怎么办？* 分片状态在 Redis 持久化，重启可恢复；合并操作支持幂等可重试。
- *重新设计你怎么改？* 加秒传（先比 MD5 直接返回已有文件）和直传 OSS（减轻服务器压力）。
